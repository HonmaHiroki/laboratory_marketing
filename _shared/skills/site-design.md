# スキル：WEBサイト構築設計（SEO対応）

**使用するエージェント：** campaign-executor（全グループ）/ content-strategist  
**入力：** サイトの目的・ターゲット・CEP・必要なコンテンツ種別  
**出力：** サイトマップ・URL設計・画面設計・ガイドライン・システム設計書

---

## 前提：なぜSEOを意識したサイト設計が必要か

WEBサイトはターゲットに訪問してもらって初めて価値を生む。検索流入は能動的に情報を探すユーザーとの相性が良く、購買行動の中心的な情報収集手段である。

| データ | 内容 |
|-------|------|
| 国内調査 | 購入前情報収集で最も利用される媒体は「インターネット検索」62.4% |
| 世界調査 | 消費者の54%が購入前情報収集で検索エンジンを最も多く利用 |
| AI普及 | 消費者の44%が購入決定前にAIチャットボットで商品情報を検索することに関心 |
| 比較検討フェーズ | オンライン利用が前年比+6.4%で増加中 |

**結論：** 情報収集経路は多様化（SNS・動画・生成AI）しているが、検索は比較検討フェーズで依然最も使われる。検索経由の流入を取りに行くサイト設計の重要性は変わらず高い。

---

## 全体ステップ

```
STEP 1：サイト・コンテンツ種別の確認
STEP 2：戦略設計（目的・目標）
STEP 3：要件設計（機能仕様・コンテンツ仕様）
STEP 4：構造設計（内部リンク設計）
STEP 5：骨格設計（画面設計・ビジュアル・マークアップガイドライン）
STEP 6：システム設計（CMS選定・設計）
```

## STEP 1：サイト・コンテンツ種別の確認

### サイトの種類（10種類）

| サイト種別 | 目的 | 検索流入 |
|----------|------|---------|
| コーポレートサイト | 企業情報発信・イメージ向上・信頼獲得 | 必要 |
| ブランドサイト | ブランド認知 | 必要 |
| リクルートサイト | 採用促進 | 必要 |
| ECサイト | 商品・サービス販売 | 必要 |
| メディアサイト | ブランド認知・ナーチャリング | 必要 |
| プロモーションサイト | ブランド認知・商品販売 | 必要 |
| 会員サイト | LTVの向上 | **不要**（限定公開） |
| ランディングページ | 商品・サービス販売 | 必要 |
| イントラネット | 社員向け情報 | **不要**（限定公開） |
| Webアプリケーション | 特定の目的を達成するツール | 場合による |

※会員サイト・イントラネットはSEO意識不要だが設計手法に大きな差異はない。

### コンテンツの種類（16種類）

```
会社概要 / 最新情報 / 事業内容 / サービス・商品説明 / 実績紹介 /
よくある質問 / コラム / キャンペーン / 採用情報 / 代表メッセージ /
アクセス / 免責事項 / IR / CSR / お問い合わせフォーム / 購入フォーム
```

サイト種別の目的と照らし合わせ、必要なコンテンツ種別を取捨選択する。

## STEP 3：要件設計

### 3-1. 機能仕様

| 要件区分 | 内容 |
|---------|------|
| **フロントエンド要件** | ターゲットブラウザ・OSなどユーザーが直接目にする表層部分 |
| **バックエンド要件** | 機能選定・セキュリティ・運用保守・サーバー・ドメイン等のシステム領域 |
| **データ要件** | アクセスログ等のデータを収集・保存するデータレイク（DL）の整理 |

### 3-2. コンテンツ仕様

**①コンテンツの洗い出し（ロジックツリー）**

TOPページを起点に、上下階層関係と同一階層をMECEで分解する。
分解軸は以下3つから選択（複数組み合わせ可）：

| 軸 | 適している場面 |
|----|-------------|
| **WHAT** | 頂点カテゴリーの中身を分解。どのサイトにも適用可能 |
| **WHY** | 課題を頂点に要因を分解。問題解決型メディアに適している |
| **HOW** | 目的・目標を頂点に手段・プロセスを分解。FAQ・ヘルプ構成に有効 |

MECE確認にはラッコキーワードで関連語を調べ、漏れがないか検証する。

**②コンテンツの整理（サイトマップ化）**

| フレームワーク | 内容 |
|-------------|------|
| **LATCH** | Location（場所）/ Alphabet（アルファベット）/ Time（時間）/ Category（カテゴリ）/ Hierarchy（階層）の5分類 |
| **カードソーティング** | LATCH等で分類した方法がユーザーにとって有効かを調査で検証 |

**③URLのラベリング**

```
□ ページ内容と関連するキーワードを含める
□ 極力短くする
□ 複数単語の組み合わせはハイフンで区切る
□ 英単語は小文字に統一する
```

## STEP 4：構造設計（内部リンク設計）

クローラーにコンテンツを認識してもらうために内部リンクは必須。
**縦横に網羅的に**設置することが重要。

### リンクの方向性

```
トップ階層との縦リンク：どのコンテンツからもTOPへ戻れる
上下階層間の縦リンク：現在地の上下階層へのリンク
同一カテゴリ間の横リンク：同カテゴリ内のコンテンツを横でつなぐ
```

### 4種類のリンク設定方法

**①ナビゲーションリンク（ピラミッド型）**

| 種別 | 設置箇所 | 推奨リンク先 |
|-----|---------|-----------|
| グローバルナビ（ヘッダー） | 全ページ共通・最上部 | 第2階層のカテゴリ等、主要リンク |
| グローバルナビ（フッター） | 全ページ共通・最下部 | 重要度がそこまで高くないリンク |
| ローカルナビ | 特定ページのみ（サイドナビ等） | 特定カテゴリ内の遷移 |

ピラミッド型：TOPページを頂点に階層構造を作る。上階層ほどリンクが集中しリンクジュースが集まる。
**下層からリンクジュースを上げ、上階層でビッグワード・下階層でロングテールワードを拾う構造にする。**

**②パンくずリスト**
TOPから現在ページまでの階層を表す（上下階層間の縦リンク）。

**③コンテンツリンク（トピッククラスター型）**

```
ピラーページ（ビッグ/ミドルワードを狙う）
    ↑ リンクジュースが集中
クラスターページ群（ロングテールワードを拾う）
```

ピラーページとクラスターページを構成することで、多岐にわたる検索意図をトピッククラスター全体で充足する。

**④HTMLサイトマップ**
全コンテンツ・カテゴリをMECEに網羅したリンク集。ページ回遊を促進する。

## STEP 5：骨格設計

### 5-1. 画面設計（Atomic Design）

UIコンポーネントを5階層で設計し、ガイドライン・システム設計に一貫性を持たせる。

```
Atoms（原子）→ Molecules（分子）→ Organisms（有機体）→ Templates（テンプレート）→ Pages（ページ）
```

**Atoms（最小要素・4分類）**

| 分類 | 内容 | 例 |
|-----|------|---|
| プラットフォームのデフォルトUI | HTML標準提供要素 | テキスト・テキストインプット・ボタン |
| デファクトスタンダードUI | 多くのアプリで使われ瞬時に理解されるUI | バルーン・バッジ・カード |
| レイアウト・パターン | コンテンツに依存しない構造 | グリッド・聖杯レイアウト・メディアオブジェクト |
| セマンティックなデザイン要素 | 意味を持つデザイン | 見出し・本文・コンテンツ画像・アイコン・区切り線・アニメーション |

**Molecules（分子）**：Atomsを組み合わせ具体的な動機に応える。例：検索フォーム（テキストインプット+ボタン）

**Organisms（有機体）**：それ自体でコンテンツとして独立成立する単位。

**Templates（テンプレート）**：Organisms/Molecules/Atomsを組み合わせたページ雛形。画面全体のレイアウト設計が完了する。

**Pages（ページ）**：Templatesを使って実際にユーザーへ提供される。

**プロトタイプ作成時の確認observation：**
```
□ ナビゲーションデザイン：グローバルナビ・サイト内リンク・優先要素へのアクセス方法
□ 情報デザイン：アイコン・タグ付けで現在地・行先を把握しやすくする配慮
```

---

### 5-2. ビジュアルガイドライン設計

| 要素 | 内容 |
|-----|------|
| **カラーパレット** | メイン・アクセント・バックグラウンドの3色を最低限定義（RGBで設定） |
| **タイポグラフィ・フォント** | タイトル・サブタイトル・見出し・本文ごとにフォント・サイズを定義 |
| **UI** | 画面設計のUIに、サイト内で使用する際のルールを定義 |

※ブランド構築フェーズでビジュアルアイデンティティを定義済みの場合は、ロゴ・ブランドカラーを引き継ぐ。

---

### 5-3. マークアップガイドライン設計

**①セマンティックなマークアップ（構造化タグ）**

文章構造を明示化するHTML5タグ：

| タグ | 役割 |
|-----|------|
| `<header>` | ヘッダー・導入・ナビゲーション補助（セクションに付随） |
| `<section>` | 見出し+本文+補足情報で構成される文章のまとまり |
| `<article>` | 独立した記事・コンテンツ（他から独立して成立する） |
| `<aside>` | 補足情報（本文と関連するが区別すべき情報） |
| `<nav>` | ナビゲーション（グローバルナビ・パンくず等） |
| `<footer>` | フッター・著作権情報・関連リンク |
| `<div>` | 見出しをつけられない意味のまとまりのみ使用（構造タグではない） |

判定基準：他セクションから独立して成り立つ → `<article>` / 成り立たない → `<section>`

**②構造化マークアップ（JSON-LD推奨）**

| 形式 | 特徴 |
|-----|------|
| Microdata | HTML5標準。データ一致しやすいがソースが煩雑 |
| RDFaLite | XHTML向け。Microdataと同様の特性 |
| **JSON-LD（推奨）** | ソースと分離可能・メンテナンス性が高い。Googleが推奨 |

**主要な構造化マークアップ種別（コンテンツ種別との対応）**

| コンテンツ種別 | スキーマ | 用途 |
|-------------|---------|------|
| 記事・コラム | Article | ニュース・ブログ記事のリッチリザルト |
| FAQ | FAQPage | 質問と回答の一覧 |
| 商品ページ | Product | 価格・在庫・クチコミ評価 |
| クチコミ | Review / AggregateRating | クチコミ抜粋・評価スコア |
| ハウツー記事 | HowTo | タスク完了の手順（動画・画像・テキスト対応） |
| 店舗情報 | LocalBusiness | 営業時間・評価・予約アクション |
| ロゴ | Logo | 検索結果・ナレッジパネルでのロゴ表示 |
| 階層位置 | BreadcrumbList | パンくずリスト |
| Q&A | QAPage | 質問→回答形式のページ |
| レシピ | Recipe | レシピのリッチリザルト |
| イベント | Event | 開催日時・場所のリッチリザルト |
| 求人 | JobPosting | 求人検索結果への表示 |

ツール活用：「構造化データマークアップ支援ツール」で記述支援、「リッチリザルトテスト」で記述エラー確認。

**③パフォーマンスバジェット**

ページ表示速度はSEO評価観点の1つ。リソース容量に上限を設定する。

| 項目 | バジェット |
|-----|----------|
| HTML | 10KB |
| 画像 | 30KB |
| CSS | 10KB |
| JS | 100KB |
| フォント | 20KB |
| **合計** | **170KB**（3G回線でもストレスなく表示できる目安） |

※ページごとに画像・JS量は異なるため、170KB内での配分はページ個別に調整する。

## STEP 6：システム設計

### 6-1. 開発手段の選定

| 開発手段 | 自由度 | コスト | 特徴 |
|---------|-------|-------|------|
| スクラッチ開発 | 高 | 高 | 機能を独自に一から構築 |
| パッケージ開発 | 中 | 中 | 標準機能搭載パッケージを使用 |
| オープンソース開発 | 中 | 中 | 標準機能搭載のOSSを使用 |
| クラウドサービス利用 | 低 | 低 | サービス提供機能をそのまま利用 |

### 6-2. CMS選定

CMSシェア（世界）：WordPressが最多、次いでShopify。
**日本国内ではWordPressが8割以上**、2番目はAdobe Systems系。

スクラッチ開発の場合はCMS選定不要。パッケージ/OSS/クラウドサービスを使う場合は選定が必要。
